Behavior Language for Embedded Systems with Software (BLESS)

The Architecture Analysis and Design Language (AADL) expresses systems-engineering models of embedded computer systems. AADL models computer systems that control machines, particularly safety-critical machines such as aircraft, medical devices, and automobiles. More can be found about AADL at AADL.info.

AADL supports extension of the core architecture language with definition of properties and annex sublanguages. AADL properties define values for architectural entities. AADL annex subclauses attach text in a domain-specific language (DSL) to AADL components. AADL annex libraries contain DSL text not bound to any particular component.

BLESS defines five annex sublanguages:

Unit
defining globally-visible units
Typedef
defining globally-visible data types
Assertion
defining globally-visible temporal logic predicates
Action
defining behavior of passive subprograms
BLESS
defining behavior of active components (i.e. threads and devices) which can initiate and respond to events

The BLESS Editor is a plug-in to the Open-Source AADL Tool Environment (OSATE), which is a plug-in to Eclipse. If you are reading this, you have both OSATE and BLESS installed. The BLESS editor checks grammar, highlights grammar, resolves names, and checks types+units. (BLESS v3 unifies type and unit checking.)

The BLESS Proof Assistant transforms Action and BLESS annex subclauses into deductive proofs given human guidance. Action and BLESS behaviors must have non-executed assertions interspersed between active constructs which express what's true about the program or system at particular moments in time.

Many examples of AADL models having BLESS specifications, behaviors, and proofs can be found at multitude.net.

The BLESS language reference manual (LRM) can be found at BLESS Manual.
